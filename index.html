3D Gaussian Splatting

From Discrete Points to Continuous Volumes

1. Introduction (The Hook)

Why move away from Point Clouds or NeRFs?

  • The Problem: Point clouds are discrete and sparse; NeRFs are continuous but computationally expensive to query.
  • The Solution: Represent the scene as a collection of millions of semi-transparent 3D Gaussian kernels.
  • The Intuition: Imagine "painting" a scene with fuzzy, translucent stickers that have specific shapes, colors, and orientations.

2. Initialization (The Seeds / SfM)

Before optimizing any 3D Gaussians, we need an initial guess of where the object is in space.

  • Multiple Perspectives: We capture images of the object from various angles around it.
  • Structure-from-Motion (SfM): Software like COLMAP analyzes visual features across these images to triangulate points in 3D space.
  • Sparse Point Cloud: The result is a cloud of floating points. These points serve as the initial centers for our Gaussian splats.

Why are SfM Point Clouds "Sparse"?

COLMAP only triangulates points where it finds highly confident feature matches (like sharp edges or textures). It leaves smooth walls, skies, or low-texture areas completely blank, resulting in a cloud missing vast amounts of surface data.

The 3DGS Advantage ("Splatting"): This is why we don't just stop at point clouds. 3DGS uses these sparse points as "seeds" and physically expands (splats) them into overlapping, fuzzy volumetric ellipsoids. Through optimization, it grows them into a dense, gapless, and continuous scene.

3. The Anatomy of a Gaussian (Geometry & Appearance)

Let's define the primitive that makes everything work: the 3D Gaussian Kernel.

The Probability Density Function (PDF)

Why use a PDF for a "Splat"?

By defining each splat as a Probability Density Function, we move from discrete points to a continuous and differentiable volume. Here is why this is crucial:

  1. Mathematical Definition: The Normal Distribution

    Each Gaussian kernel is essentially a 3D Normal Distribution. The value f(x) describes the "influence" of that splat at any point x in space.
    — μ (Mean): Represents the center of the splat.
    — Σ (Covariance): Defines the shape, scale, and orientation.

  2. From "Hard Edge" to "Soft Contribution"

    Unlike traditional meshes or points with hard boundaries, a PDF provides spatial continuity. Every point in space is "influenced" by every Gaussian to some degree, though it decays exponentially with distance.

    Most importantly, it is Differentiable. Because the PDF is smooth, we can use gradient descent to optimize the splat's properties. We can calculate exactly how a tiny shift in position or shape affects the final image.

  3. Rendering Physics: Learned Opacity (α)

    In rendering, this PDF represents the spatial extent of the Gaussian's contribution. It is scaled by a learned opacity factor α to determine precisely how much light it blocks at each point in space:

Decomposing the Covariance (Σ)

While a 3D Gaussian is defined by a 3x3 symmetric matrix (6 degrees of freedom), optimizing these values directly is difficult because the matrix must remain Positive Definite (all eigenvalues > 0) to represent a valid, non-degenerate ellipsoid.

Instead, we decompose the covariance into Scaling (S) and Rotation (R, via Quaternions). This parametrization allows the optimizer to move freely in space while ensuring the resulting matrix always lands in the valid positive-definite subset:

3.2 View-Dependent Color (Spherical Harmonics)

3DGS doesn't just store a single RGB value. Instead, it uses Spherical Harmonics (SH) so that a Gaussian can look different from different angles—reproducing highlights and reflections.

The Intuition: Imagine each Gaussian "sticker" is covered in a holographic material. As you walk around it, the color shifts and glitters based on your perspective.

4. The Forward Pass (Projection & Tile-based Rasterization)

A. Projection (3D to 2D Splatting)

How a 3D ellipsoid becomes a 2D "splat" on the screen.

B. Fast Tile-Based Rasterization

This is the magic that enables real-time performance. Unlike NeRF's expensive per-ray MLP queries, 3DGS uses a high-speed tile-based rasterization pipeline on the GPU:

  1. Instancing & Culling: The screen is divided into 16x16 Tiles. Each Gaussian identifies which tiles it touches and creates an "instance" for each, allowing parallel processing.
  2. Radix Sort: Within each tile, the relevant Gaussians are **sorted by depth**. This precise ordering is the prerequisite for correct alpha blending.
  3. Parallel Alpha Blending: For every pixel in the tile, we traverse the sorted list and compute the final color using the Volume Rendering Equation:

The Physics of Blending

Transmittance (Ti): This term represents the probability that light can pass through all preceding Gaussians to reach the camera. If a Gaussian in front is fully opaque, Ti drops to zero, making objects behind it invisible.

5. The Backward Pass (Optimization & Densification)

We've discussed initialization and rendering, but what makes the Gaussians actually fit the photos? That's the training loop using Stochastic Gradient Descent (SGD).

A. The Loss Function

Every iteration, we compare the rendered photo (I_render) with the actual ground truth photo from the dataset (I_gt). The difference determines our error.

We combine L1 Loss (for absolute pixel differences) with D-SSIM (Structural Similarity Index, to preserve sharp edges and local structures).

B. Adaptive Density Control (Densification)

This is the core contribution of the 3DGS paper. While optimizing the shape and color of Gaussians, the algorithm also decides to add or remove them based on necessity:

Clone (Under-reconstruction)

→
When a Gaussian is too small but covers an area of high error.
We clone it and move it slightly precisely in the direction of the error gradient to fill the gap.

Split (Over-reconstruction)

→
When a Gaussian is too large and blurring high-frequency details.
We split it into two smaller Gaussians (scaling them down by 1.6x) to capture sharper features.

C. Pruning

Periodically, the algorithm cleans up the scene by removing Gaussians that are basically transparent (Opacity α drops near zero) or those that have become unreasonably large and are "leaking" into empty space. This keeps the rendering fast and the file size manageable.

7. From Splats to Mesh (Marching Cubes)

3D Gaussians are volumetric blobs—great for photorealistic rendering, but difficult for traditional physics or 3D engines to handle. To get a standard triangle mesh, we use a pipeline often referred to as SDF-to-SURFACE.

The SDF-to-SURFACE Pipeline

  1. Virtual Rendering: Projections from "virtual cameras" generate high-resolution Depth and Opacity maps from the trained Gaussian field.
  2. TSDF Fusion (Weighted Integration):

    For each voxel in a 3D grid, we calculate the signed distance to the observed depth surface along the camera ray. These distances are then weighted and averaged across multiple views to filter out noise and create a coherent field.

    The "T" in TSDF (Truncated): To save memory and compute, we only store distance values within a narrow band near the surface. Outside this band, the value is truncated.

  3. Finding the 0-Level Set: We identify the precise boundary where the field transitions from Positive (Outside) to Negative (Inside). * Note: The +/- convention is a common standard (e.g., nvblox, KinectFusion), but some implementations may swap them.
  4. Marching Cubes (MC): The final geometric mesh is extracted by constructing triangles along this 0-level boundary.

Beyond TSDF: Modern Approaches

While Depth-to-TSDF is a mainstream route, newer research directly enforces surface geometry during training:

  • SuGaR / 2DGS: Uses regularization to force Gaussians to be flat and align with the surface.
  • GOF: Defines a transparency-based level set directly on the Gaussian representation to bypass the need for virtual rendering.
Technical Insight: Visualization vs. Extraction

Modern engines like nvblox can show the TSDF field in real-time, but they do so using Ray Casting (tracing a ray through the voxels until it hits a zero-crossing). This is only for display—extracting the actual geometry for other tools still requires Marching Cubes.
